# Single Upload Complete Flow

## Overview

Single upload addresses this scenario:

- Client requests a temporary upload URL
- API creates local metadata
- Client uploads directly to S3 using PUT
- Client confirms completion
- API validates against S3
- Database marks file as completed

### Involved Tables

Only these tables from the schema are involved:

- `nodes`
- `files`
- `file_upload_sessions`
- `file_upload_events`

Note: `multipart_uploads` and `multipart_upload_parts` are exclusive to multipart flow.

---

## 1. Initial Expected State

When a client initiates a single upload, the database should look like this:

```
nodes.type = 'file'
files.status = 'pending'
file_upload_sessions.upload_mode = 'single'
file_upload_sessions.status = 'pending'
```

### Why Not "Uploading"?

Single upload doesn't track an active session on S3. The client might:
- Use the URL immediately
- Use it 10 minutes later
- Never use it
- Lose connection
- Close the browser

The realistic state machine is:
```
pending → completed
pending → failed
pending → canceled
pending → expired
```

The "uploading" state makes more sense for multipart, where a live S3 session exists.

---

## 2. Initialization Endpoint

Recommended endpoint:

```http
POST /files/uploads/single
Content-Type: application/json

{
  "parentId": "7f0e8c5d-0f5e-4f3e-9a31-7f2f582c3b25",
  "fileName": "document.pdf",
  "mimeType": "application/pdf",
  "size": 248312
}
```

`parentId` can be `null` if uploads to root are allowed.

---

## 3. Initial Validations (C#)

Before creating any metadata, validate:

- `fileName` is not empty
- `mimeType` is not empty
- `size > 0`
- `size <= single_upload_max_size` (from config)
- `parentId` exists if provided
- `parentId` belongs to an active folder

**Configuration Example:**
```
single_upload_max_size = 100 MiB
```

If the file exceeds this limit, route it through multipart upload instead.

---

## 4. Generating Identifiers

Best practice: **Generate nodeId in C# before inserting**, not letting PostgreSQL auto-generate:

```csharp
var nodeId = Guid.NewGuid();
```

Why?
- You need `object_key` before creating the presigned URL
- Avoids intermediate states between insert and URL generation
- Cleaner workflow: generate IDs → create metadata → generate URL

---

## 5. Object Key Generation

The object key must be:
- Stable and deterministic
- Not dependent on the original filename
- Safe from path traversal or special characters

**Recommended Format:**
```
files/{nodeId}/original{extension}
```

**Example:**
```
files/2dd0168e-52b6-4da7-96fc-3b48c5fd2878/original.pdf
```

**Avoid:**
```
uploads/document.pdf  ❌ (collision/manipulation risk)
```

---

## 6. Calculating Expiration

C# calculates:

```csharp
var expiresAt = DateTime.UtcNow.Add(_fileConfig.UploadExpiration);
```

This same `expiresAt` is used for:
1. The presigned URL signature
2. `file_upload_sessions.expires_at`

They **must match**. Don't let PostgreSQL calculate a different expiration.

---

## 7. Creating Metadata in Database

Call a database function (future):

```sql
create_upload_session(
  nodeId,
  parentId,
  fileName,
  mimeType,
  size,
  objectKey,
  expiresAt
)
```

This should atomically insert:

### Into `nodes`
- `id` = nodeId
- `parent_id` = parentId
- `name` = normalized fileName
- `type` = 'file'

### Into `files`
- `node_id` = nodeId
- `mime_type` = mimeType
- `size` = size
- `object_key` = objectKey
- `status` = 'pending'

### Into `file_upload_sessions`
- `id` = sessionId (generated by DB)
- `node_id` = nodeId
- `upload_mode` = 'single'
- `status` = 'pending'
- `expires_at` = expiresAt

### Into `file_upload_events`
- `node_id` = nodeId
- `session_id` = sessionId
- `from_status` = null
- `to_status` = 'pending'
- `reason` = 'Single upload session created'
- `metadata` = { ... }

**All in a single transaction.**

---

## 8. Generating Presigned URL

After creating metadata, generate the upload URL:

```csharp
var uploadUrl = _storageService.GenerateUploadUrl(
    objectKey,
    mimeType,
    expiresAt);
```

Requirements:
- HTTP verb: `PUT`
- `Content-Type`: mimeType
- Expires: expiresAt

**Critical:** If you sign the URL with `Content-Type`, the client must send exactly that header when uploading.

---

## 9. Client Response

Return to client:

```json
{
  "nodeId": "2dd0168e-52b6-4da7-96fc-3b48c5fd2878",
  "sessionId": "bd1f5665-b89b-4e2a-bbf4-7249b59e093a",
  "uploadMode": "single",
  "uploadUrl": "https://s3.region.amazonaws.com/bucket/...",
  "expiresAt": "2026-05-16T18:30:00Z"
}
```

Optional: You can omit `objectKey` in production (only include for debugging).

---

## 10. Client Uploads File

The client performs:

```http
PUT {uploadUrl}
Content-Type: application/pdf

[binary file content]
```

The API does **not** receive the file. It goes directly to S3.

---

## 11. Upload Confirmation

After uploading, the client calls:

```http
POST /files/uploads/{sessionId}/confirm
```

Recommendation: Use `sessionId` in the endpoint (not just `nodeId`) to make the upload session explicit.

---

## 12. Confirmation: C# Validations

Before marking as completed, validate:

- Session exists
- `session.upload_mode = 'single'`
- `session.status = 'pending'`
- `session.expires_at > now()`
- File exists
- `file.status = 'pending'`

If the session is expired, **reject confirmation**.

---

## 13. Real Validation Against S3

**Critical step:** Before marking completed, perform `HEAD Object`:

```csharp
var response = await _s3Client.GetObjectMetadataAsync(
    new GetObjectMetadataRequest
    {
        BucketName = _bucketName,
        Key = objectKey
    });
```

Validate:
- ✅ Object exists
- ✅ `ContentLength == files.size`
- ✅ `ContentType` compatible with `files.mime_type`

**If any validation fails:**
- Do **not** mark as completed
- Return error to client
- Mark as failed if it's a permanent issue

**Never mark as successful just because the client said "I uploaded".**

---

## 14. Completing Upload in Database

If S3 validation passes, call:

```sql
complete_upload_session(
  sessionId,
  contentLength,
  contentType,
  etag,
  metadata)
```

Database must atomically execute:

### Update `file_upload_sessions`
- `status` = 'completed'

### Update `files`
- `status` = 'completed'

### Insert `file_upload_events`
- `from_status` = 'pending'
- `to_status` = 'completed'
- `reason` = 'Upload completed'
- `metadata` = { contentLength, contentType, etag }

---

## 15. Confirmation Response

Option A (minimal):
```json
{
  "message": "File upload confirmed successfully."
}
```

Option B (recommended):
```json
{
  "nodeId": "2dd0168e-52b6-4da7-96fc-3b48c5fd2878",
  "name": "document.pdf",
  "mimeType": "application/pdf",
  "size": 248312,
  "status": "completed"
}
```

Option B is better if the frontend needs to refresh file state.

---

## 16. Expiration Flow

A worker runs periodically:

```sql
SELECT * FROM file_upload_sessions
WHERE status IN ('pending')
  AND expires_at <= now()
LIMIT 100;
```

For each expired session:

### Database Operations
- Mark `file_upload_sessions.status` = 'expired'
- Mark `files.status` = 'expired'
- Insert `file_upload_events` (reason: 'Upload expired')

### Optional: Storage Cleanup
If the object exists in S3, decide:

**Option A: Strict (Recommended for v1)**
- Attempt `DeleteObject(objectKey)` in S3
- Log failures but don't fail the expiration process

**Option B: Recovery Option**
- Leave the object in S3
- Allow administrative recovery later

---

## 17. Cancellation Flow

Endpoint:

```http
POST /files/uploads/{sessionId}/cancel
```

### C# Logic
1. Query the session
2. If status is `pending`:
   - Call `finish_upload_session(sessionId, 'canceled', reason)`
3. Optionally attempt `DeleteObject(objectKey)` in S3

### Database Operations
- `file_upload_sessions.status` = 'canceled'
- `files.status` = 'canceled'
- Insert `file_upload_events` (reason: 'Upload canceled')

---

## 18. Failure Flow

Failure can be caused by:
- S3 HEAD doesn't find the object
- Size mismatch
- MIME type mismatch
- Permanent storage error
- Cancellation issues

### C# Logic
Call:
```csharp
finish_upload_session(sessionId, "failed", reason, metadata);
```

### Database Operations
- `file_upload_sessions.status` = 'failed'
- `files.status` = 'failed'
- Insert `file_upload_events` (reason: failure reason)

---

## 19. State Machine

```
pending
  ├── completed ✓
  ├── failed
  ├── canceled
  └── expired
```

### Forbidden Transitions
```
completed → pending     ❌
completed → failed      ❌
expired → completed     ❌
canceled → completed    ❌
failed → completed      ❌
```

To retry, create a **new session**, not resurrection of an old one.

---

## 20. Retry Strategy

If an upload fails, expires, or is canceled, allow retry:

### Recommended Approach
- **Same** node/file
- **New** file_upload_session
- **Same** object_key

Before retrying, ensure any previous S3 object is cleaned up or will be overwritten.

---

## 21. Complete Flow Summary

### INITIALIZATION

```
Client → API:
  POST /files/uploads/single

C#:
  - Validate request
  - Validate size for single upload
  - Generate nodeId
  - Generate objectKey
  - Calculate expiresAt

Database:
  - Create node + file
  - Create file_upload_session (single, pending)
  - Create audit event

C#:
  - Generate presigned PUT URL
  - Return to client
```

### UPLOAD

```
Client → S3:
  PUT {uploadUrl}
  [file content]
```

### CONFIRMATION

```
Client → API:
  POST /files/uploads/{sessionId}/confirm

C#:
  - Validate session/file state
  - Validate expiration
  - HEAD Object in S3
  - Validate size/mime type
  - Call complete_upload_session()

Database:
  - Mark session as completed
  - Mark file as completed
  - Insert audit event

API → Client:
  200 OK + file metadata
```

---

## 22. Critical Design Decision

**NEVER mark a file as completed without verifying it exists in S3 with correct metadata.**

If you skip the S3 HEAD Object validation, your database will say:
```
status = 'completed'
```

While in S3:
- The object doesn't exist, OR
- The object has the wrong size, OR
- The object has the wrong MIME type

This creates data integrity issues.

**Always validate against S3 before updating the database.**

---

## 23. Summary of Best Practices

| Practice | Reason |
|----------|--------|
| Generate nodeId in C# | Enables object_key before DB insert |
| Use sessionId in endpoints | Makes upload session explicit |
| Validate S3 before DB update | Ensures data integrity |
| Same expiresAt for URL & session | Prevents expiration mismatches |
| Atomic DB transactions | Prevents partial state |
| Reject retry of same session | Forces new session creation |
| Object key without filename | Prevents collision/manipulation |
